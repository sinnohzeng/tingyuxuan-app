## 一、产品概述

### 1.1 产品定位

听语轩是一款拥有自主知识产权的 AI 驱动智能语音输入工具，面向个人用户和企业内部推广。不同于传统的语音转文字（STT）工具，听语轩的核心差异化在于：在语音识别的基础上叠加了大语言模型（LLM）的后处理能力，能将用户自然、口语化的表达自动整理为结构清晰、表达规范的书面文字。

**一句话定义**：听语轩 = 语音识别（STT） + 大模型智能润色（LLM Post-Processing） + 系统级全局输入

**与 Typeless 的关键差异**：听语轩是完全自主开发的产品，支持用户灵活配置自己的 STT API 和 LLM API（包括私有部署的模型），不依赖任何第三方 SaaS 服务的账号体系。

### 1.2 产品命名

当前暂定名称为「听语轩」：

- **听**：聆听——产品的核心输入方式
- **语**：语言——连接口语与书面语的桥梁
- **轩**：高雅、开阔——寓意将粗糙的口语提升为精致的文字

**备选命名建议**（供团队讨论）：

| 候选名称 | 寓意 | 英文名建议 | 优劣分析 |
|----------|------|-----------|---------|
| **听语轩** | 聆听语言的高雅空间 | TingYuXuan / Listencraft | 文化感强，有品牌辨识度；但三字名在国际化中较长 |
| **听书** | 听你说、书写出来 | TingShu / Scribevoice | 简洁有力；但容易和"有声书"概念混淆 |
| **语流** | 语言如流水般自然输出 | YuFlow / VoiceFlow | 现代感强；但 VoiceFlow 已是已有产品名 |
| **述辑** | 口述 + 编辑 | ShuJi / Dictcraft | 精准表达产品功能；但略显生僻 |
| **言润** | 言语润色 | YanRun / VoicePolish | 直接点明 LLM 润色能力；朗朗上口 |
| **声笔** | 声音是你的笔 | ShengBi / VoicePen | 比喻直观；容易理解和传播 |
| **妙语** | 妙笔生花的语音版 | MiaoYu / CleverVoice | 有趣味性；暗示 AI 让语言变"妙" |

**命名建议**：从品牌传播角度看，「听语轩」文化气质好但国际化偏弱。如果产品主要面向国内市场和内部推广，听语轩是很好的选择。如果后续有出海计划，建议同时确定一个简短的英文品牌名（如 VoicePen、Dictcraft 等），不必是中文名的直译。

### 1.3 核心价值

- **速度**：语音输入速度约 220 WPM，是键盘打字（45 WPM）的近 5 倍
- **质量**：自动去除口语填充词、修正口误、整理排版，输出即发送级文本
- **全局性**：在任何应用中均可使用，从 Slack、微信、钉钉到邮件客户端、代码编辑器
- **隐私**：零数据留存策略，语音数据不存储、不用于训练模型
- **灵活性**：用户可配置自己的 STT API 和 LLM API，不绑定特定服务商

## 二、核心技术架构

### 2.1 处理管线（Pipeline）

听语轩采用三阶段管线架构：

```
┌──────────────────────────────────────────────────────────────────────┐
│                        用户触发快捷键                                 │
│                              │                                       │
│                              ▼                                       │
│  ┌──────────────┐     ┌──────────────────┐     ┌─────────────┐      │
│  │  麦克风音频   │ ──→ │  本地音频录制     │ ──→ │  本地缓存     │      │
│  │  （实时流）    │     │  + 实时波形显示   │     │  WAV/WebM    │      │
│  └──────────────┘     └──────────────────┘     └──────┬──────┘      │
│                                                        │             │
│                        用户停止录音                      │             │
│                              │                          │             │
│                              ▼                          ▼             │
│                     ┌──────────────────┐     ┌─────────────┐         │
│       阶段 1        │  语音识别（STT）   │ ←── │  上传音频文件 │         │
│                     │  Whisper / 自定义  │     └─────────────┘         │
│                     └────────┬─────────┘                              │
│                              │                                       │
│                              ▼                                       │
│                     ┌──────────────────┐                              │
│       阶段 2        │  LLM 后处理层     │                              │
│                     │  - 去填充词       │     ┌──────────────────┐    │
│                     │  - 修正口误       │ ←── │  上下文感知信息    │    │
│                     │  - 整理排版       │     │  (当前App/选中文本)│    │
│                     │  - 语境适配       │     └──────────────────┘    │
│                     │  - 翻译（可选）    │                              │
│                     │  - AI指令（可选）  │                              │
│                     └────────┬─────────┘                              │
│                              │                                       │
│                              ▼                                       │
│       阶段 3        ┌─────────────────┐                               │
│                     │  输出注入层       │                               │
│                     │  模拟键盘输入     │                               │
│                     │  插入目标应用     │                               │
│                     └─────────────────┘                               │
└──────────────────────────────────────────────────────────────────────┘
```

**阶段 0 — 本地音频录制与缓存（关键！）**：
- 按下快捷键后，立即在本地开始录制音频
- 音频实时写入本地临时文件（WAV 或 WebM 格式）
- 同时驱动 UI 层的实时波形显示
- **这一步是纯本地的，不依赖网络** —— 这是兜底机制的基础

**阶段 1 — 语音识别（STT）**：
- 将本地缓存的音频文件上传到 STT 服务
- 支持 100+ 种语言的自动检测
- 支持多语言混说（如中英混杂）
- 需要联网处理（非纯本地），但音频已安全缓存在本地

**阶段 2 — LLM 智能后处理**：
- 去除填充词（um, uh, 嗯, 那个...）
- 检测并消除重复表述
- 识别口误自动修正（只保留最终意图）
- 自动排版（列表、段落结构化）
- 根据当前应用上下文调整语气
- 翻译模式下执行翻译
- AI 模式下执行指令（搜索、生成等）

**阶段 3 — 输出注入**：
- 通过系统级模拟键盘输入，将最终文本注入到当前光标所在位置
- 无需目标应用做任何适配

### 2.2 上下文感知系统

听语轩的一个关键技术点是**应用上下文感知**：

- 通过系统辅助功能 API（macOS Accessibility API / Windows UI Automation）识别当前活跃应用
- 根据应用类型动态调整 LLM 的 prompt/处理策略：
  - **WhatsApp/微信/iMessage** → 口语化、简短、非正式
  - **Gmail/Outlook/钉钉邮件** → 正式、完整句子、适当问候语
  - **Slack/钉钉/飞书** → 半正式、简洁
  - **VS Code/Cursor** → 保留技术术语、减少润色
  - **笔记应用（Notion/Obsidian）** → 结构化、保留要点、Markdown 格式
  - **终端/命令行** → 极简模式，仅转写不润色

## 三、功能模块详细规格

### 模块 A：智能听写（Dictate Mode）— 默认模式

#### A.1 基本工作流

1. 用户按下快捷键（默认 `Fn`）激活听写
2. 屏幕底部出现浮动状态条，显示实时波形动画，表示正在录音
3. 用户自然说话
4. 再次按下快捷键（或点击浮动条右侧的"✓"确认按钮）停止录音
5. 浮动条进入"处理中"状态，显示加载动画
6. 听语轩处理音频并将润色后的文本插入到当前光标位置
7. 浮动条短暂显示完成状态后自动隐藏

#### A.2 智能转述与整理功能

**填充词自动去除**：
- 自动去除：um, uh, 嗯, 那个, you know, like, basically, actually, 就是说, 然后 等
- 去除时保持语义完整性
- 支持多语言填充词识别

**口误与重复修正**：
- 当用户说"我想要……不对，我需要那个报告"时，输出"我需要那个报告"
- 识别模式：用户在一句话中间改口，系统保留后面修正过的版本
- 识别连续重复（如"这个这个这个方案"→"这个方案"）
- 识别否定修正词（"不对"、"我是说"、"应该是"、"wait, I mean"等）作为修正信号

**自动排版**：
- 当用户说"第一点……第二点……第三点……"时，自动生成有序列表：
  ```
  1. ……
  2. ……
  3. ……
  ```
- 当用户说"首先……其次……最后……"时，也能识别并结构化
- 无序列表：当用户说"有以下几点"或使用并列结构时，自动生成无序列表
- 支持 Markdown 格式输出（标题、列表、粗体等），在 v0.9.0+ 参考 Typeless 的实现
- 关键词检测：识别"标题是"、"小标题"等语音指令生成对应的 Markdown 标题

**标点与格式化**：
- 自动补充标点符号（逗号、句号、问号、感叹号、冒号等）
- 智能断句断段
- 保持用户的表达意图和语气

#### A.3 个性化学习

- **使用越多越准确**：系统学习用户的词汇习惯、表达风格、语速
- **个人词典**：用户可手动添加专有名词、品牌名、行业术语
  - 添加方式：设置 → 词典 → 添加自定义词汇
  - 系统也会自动学习频繁使用的特殊词汇
- **语气适配**：随时间学习用户偏好的写作风格
- **个性化开关**：用户可在设置中关闭个性化学习，切换为通用非个性化模式

---

### 模块 B：语音编辑（Speak to Edit Mode）

#### B.1 基本工作流

1. 用户在任意应用中**选中一段已有文本**
2. 按下听语轩快捷键（默认 `Fn`）
3. 口述编辑指令（如"把这段改成更正式的语气"）
4. 再次按下快捷键（或点击浮动条"✓"）完成
5. 听语轩将选中文本替换为编辑后的版本

#### B.2 支持的编辑操作

| 操作类型 | 示例语音指令 |
|----------|-------------|
| 语气调整 | "让这段更正式一些" / "更友好一些" / "更简洁" |
| 格式转换 | "把这段变成列表" / "改成邮件格式" |
| 内容修改 | "把日期改成下周五" / "删掉第二句话" |
| 翻译 | "翻译成英文" |
| 扩写/缩写 | "展开说明一下" / "精简到一句话" |
| 纠错 | "修正拼写错误" / "检查语法" |
| 重写 | "用不同的方式重写" / "改成更专业的表达" |
| 回复生成 | "帮我写一个积极的回复" / "委婉地拒绝" |

#### B.3 智能行为

- 当用户选中文本后按快捷键，系统检测到有选中内容，自动进入编辑模式
- 无需用户手动切换模式——系统根据"是否有选中文本"自动判断
- 编辑完成后，Typeless 的实现是可联网搜索以补充信息（v0.9.0+）

#### B.4 技术实现要点

- 需要读取当前选中文本（通过辅助功能 API 或模拟 Cmd+C 读剪贴板）
- 将选中文本 + 用户语音指令一起发送给 LLM
- LLM 返回编辑后的文本
- 通过模拟键盘输入替换选中内容

### 模块 C：翻译模式（Translation Mode）

#### C.1 基本工作流

1. 用户在设置中预先选定目标翻译语言（如英语→中文）
2. 用户按下翻译模式快捷键（默认 `Fn + Shift`）
3. 浮动条出现并显示"翻译模式"标识 + 目标语言
4. 用户用任意语言自然说话
5. 按下主快捷键（默认 `Fn`）或点击"✓"结束录音
6. 听语轩输出翻译后的文本（不是原文），直接插入光标位置

#### C.2 配置细节

**目标语言设置**：
- 路径：设置 → 语言 → 设置翻译目标语言
- 支持选择语言变体（如西班牙语-墨西哥、葡萄牙语-巴西、英语-英国等）
- 一次只能设定一个目标语言

**快捷键配置**：
- 翻译模式快捷键默认：`Fn + Shift`
- 可在设置 → 键盘快捷键中自定义
- 与主听写快捷键独立配置

#### C.3 翻译质量特点

- 输出不是直译，而是"本地化"的自然表达
- 保持原文的语气和意图
- 支持混合语言输入（如中英混说，输出为纯目标语言）
- 自动应用与听写模式相同的格式化处理（去填充词等）

---

### 模块 D：AI 助手模式（LLM Direct Call）

#### D.1 基本工作流

1. 用户按下 AI 助手快捷键（默认 `Fn + 空格`）
2. 浮动条出现并显示"AI 模式"标识
3. 用户口述指令或问题（如"帮我搜索一下最新的 React 19 特性"）
4. 按下快捷键或点击"✓"结束
5. 听语轩调用大模型处理请求，大模型可联网搜索获取最新信息
6. 结果文本直接插入到当前光标位置

#### D.2 AI 助手能力

| 能力 | 说明 |
|------|------|
| 联网搜索 | 大模型可搜索网络获取实时信息（参考 Typeless v0.9.0+） |
| 内容生成 | 根据用户描述生成文本、邮件、代码片段 |
| 问答 | 回答各类知识性问题 |
| 总结 | 对给定内容进行总结 |
| Markdown 输出 | 支持结构化的 Markdown 格式输出 |

**触发联网搜索的关键词**：当用户语音指令中包含"搜索"、"最新"、"当前"、"search"、"latest"、"current"等关键词时，系统自动启用联网搜索能力。

#### D.3 与"听写模式"的区别

| 维度 | 听写模式 | AI 助手模式 |
|------|----------|------------|
| 输入 | 用户想表达的内容本身 | 给 AI 的指令/问题 |
| 处理 | STT + 润色整理 | STT + LLM 指令执行 |
| 输出 | 用户自己话语的优化版 | AI 生成的全新内容 |
| 联网 | 不联网 | 可联网搜索 |
| 快捷键 | `Fn` | `Fn + 空格` |

## 四、交互设计详细规格

### 4.1 快捷键体系

| 功能 | macOS 默认快捷键 | Windows 默认快捷键 | 说明 |
|------|-----------------|-------------------|------|
| 听写/停止 | `Fn`（单按） | `Ctrl + Shift + D` | 按下开始，再按停止 |
| 翻译模式 | `Fn + Shift` | `Ctrl + Shift + T` | 启动翻译模式录音 |
| AI 助手 | `Fn + 空格` | `Ctrl + Shift + A` | 启动 AI 助手模式 |
| Speak to Edit | 先选中文本 → 按 `Fn` | 先选中文本 → 按 `Ctrl+Shift+D` | 自动检测选中文本进入编辑模式 |
| 取消录音 | `Esc` 或点击浮动条"✕" | `Esc` 或点击浮动条"✕" | 取消当前录音，不进行转写 |

**快捷键设计原则**：
- 所有快捷键可在设置中自定义
- macOS 基于 `Fn` 键为核心（参考 Typeless），Windows 使用 `Ctrl+Shift` 组合
- 不同模式通过组合键区分
- 避免与系统原生快捷键冲突
- macOS 上需提示用户在"系统设置 → 键盘"中将 Fn 键行为从"更改输入法"改为"无操作"

### 4.2 浮动状态条（核心 UI 组件）

这是用户与听语轩交互的主要界面元素，始终悬浮在屏幕底部居中位置。

#### 4.2.1 布局结构

```
┌─────────────────────────────────────────────────────┐
│  ✕  │          ▮▮▮▮▮ 波形可视化 ▮▮▮▮▮           │  ✓  │
│ 取消 │          （实时声纹条形图/波形图）            │ 确认 │
└─────────────────────────────────────────────────────┘
  左侧            中间区域                            右侧
```

**三栏布局**：

| 位置 | 元素 | 功能 |
|------|------|------|
| **左侧** | ✕（叉号按钮） | 取消当前录音。点击后丢弃本次录音，不进行转写处理。等同于按 Esc |
| **中间** | 声纹波形图 | 实时显示麦克风输入的音频波形，让用户直观看到声音正在被采集 |
| **右侧** | ✓（对勾按钮） | 确认/完成录音。点击后停止录音并开始 STT + LLM 处理。等同于再按一次快捷键 |

#### 4.2.2 波形可视化细节

- **样式**：垂直条形图（Bar Visualizer），不是连续曲线。每个竖条代表一个频段或时间片的音量
- **条数**：约 20-40 根竖条，均匀分布在中间区域
- **动态**：竖条高度随实时音量变化，有平滑的动画过渡
- **颜色**：使用品牌主色调（建议渐变色，如蓝→紫），未激活时为低饱和度灰色
- **刷新率**：至少 30fps，确保波形流畅

#### 4.2.3 浮动条状态机

```
┌──────────┐   按下快捷键   ┌──────────┐   按✓/再按快捷键   ┌──────────┐

     │                          │                                 │
     │                          ▼                                 ▼
```

## 五、异常处理与兜底机制（关键章节）

> **核心原则**：用户说的每一个字都不能丢。无论网络中断、服务崩溃还是应用闪退，用户的录音内容都必须有迹可循、可恢复。

### 5.1 音频本地缓存策略

这是整个兜底机制的基石：

```
录音开始
  │
  ├──→ 实时写入本地临时文件（WAV/WebM）
  │      路径：~/.tingyuxuan/cache/audio/{timestamp}_{session_id}.wav
  │
  ├──→ 同步驱动波形可视化
  │
  └──→ 录音结束后，音频文件保留在本地
         │
         ├──→ 转写成功 → 标记该音频为"已处理"
         │     音频按用户设定的保留时长保存
         │
         └──→ 转写失败 → 音频标记为"待重试"
               用户可从界面手动触发重新转写
```

**缓存规则**：
- 每次录音都生成独立的本地音频文件，命名格式：`{ISO时间戳}_{模式}_{session_id}.wav`
- 录音过程中，音频数据每 500ms flush 一次到磁盘，防止进程崩溃导致数据丢失
- 音频文件默认保留 24 小时，用户可在设置中调整
- 已成功处理的音频保留时间可以更短（如 1 小时）
- 未成功处理的音频强制保留至少 7 天，并在菜单栏显示未处理提醒

### 5.2 异常场景处理

#### 场景 1：网络不通畅/STT 请求失败

```
用户完成录音 → 上传音频到 STT 服务
                    │
                    ├── 超时（>15秒无响应）
                    ├── HTTP 错误（5xx / 网络断开）
                    └── API 额度耗尽
                         │
                         ▼
               ┌─────────────────────┐
               │  浮动条显示错误状态   │
               │  "网络连接失败"       │
               │                     │
               │  [重试]  [稍后处理]   │
               └─────────────────────┘
```

**处理流程**：
1. 浮动条从"处理中"变为"错误"状态，显示简明的错误信息
2. 提供两个按钮：
   - **「重试」**：立即重新发送同一份音频文件进行转写（因为音频已缓存在本地）
   - **「稍后处理」**：将本次录音加入"待处理队列"，用户可稍后从菜单栏手动触发
3. 自动重试策略：首次失败后自动重试 1 次（间隔 3 秒），如果仍然失败则展示手动操作界面
4. 菜单栏图标变为警告色（黄色/橙色），提示有待处理的录音

#### 场景 2：LLM 后处理失败（STT 成功但 LLM 出错）

```
STT 成功，获得原始转写文本 → 发送到 LLM
                                │
                                ├── LLM 超时
                                ├── LLM API 错误
                                └── LLM 返回异常
                                     │
                                     ▼
               ┌──────────────────────────┐
               │  "AI 润色失败"             │
               │                          │
               │  [插入原始转写]  [重试润色]  │
               └──────────────────────────┘
```

**处理流程**：
1. STT 已成功，原始转写文本已获取——这本身就是一个有价值的结果
2. 提供选项：
   - **「插入原始转写」**：将 STT 的原始转写文本直接插入光标位置（虽然有填充词、不够规范，但内容不丢失）
   - **「重试润色」**：使用已缓存的原始转写文本重新调用 LLM
3. 原始转写文本也存入本地历史记录

#### 场景 3：录音过程中应用崩溃/意外退出

```
正在录音 → 应用崩溃/闪退
              │
              ▼
    音频文件已部分写入磁盘
    （因为每 500ms flush）
              │
              ▼
    下次启动应用时自动检测
    ~/.tingyuxuan/cache/audio/ 中
    是否存在"未完成"状态的音频文件
              │
              ▼
    ┌──────────────────────────────┐
    │  "检测到上次未完成的录音"       │
    │                              │
    │  录音时间：2026-02-24 14:30   │
    │  时长：约 3 分 20 秒           │
    │                              │
    │  [立即转写]  [丢弃]  [保留]    │
    └──────────────────────────────┘
```

**处理流程**：
1. 应用启动时扫描缓存目录，找到状态为"recording"（未正常关闭）的音频文件
2. 弹出恢复对话框，显示录音时间和时长
3. 提供选项：
   - **「立即转写」**：将崩溃前录制的音频发送到 STT → LLM 处理
   - **「丢弃」**：删除该音频文件
   - **「保留」**：不处理，但保留音频文件，稍后可从历史记录中找到

#### 场景 4：麦克风未连接/无权限

```
用户按下快捷键 → 检测麦克风
                    │
                    ├── 未检测到麦克风
                    ├── 麦克风权限未授予
                    └── 麦克风被其他应用独占
                         │
                         ▼
               ┌─────────────────────────┐
               │  系统通知弹出              │
               │  "无法访问麦克风"          │
               │                         │
               │  [前往系统设置]  [知道了]   │
               └─────────────────────────┘
```

**处理流程**：
1. 在尝试录音前先检查麦克风可用性
2. 使用系统原生通知（不阻塞用户操作），提示具体原因
3. 如果是权限问题，提供跳转到系统设置的快捷方式
4. 不显示浮动录音条，避免用户误以为正在录音

#### 场景 5：长时间录音的特殊处理

> **这是用户最怕的场景：说了 5 分钟，结果内容丢了。**

**防护措施**：
- 录音过程中持续写入磁盘（每 500ms flush），即使录了 30 分钟也不会全丢
- 超过 2 分钟的录音，浮动条显示当前录音时长提醒（避免用户忘记停止）
- 超过 5 分钟的录音，系统自动对已录部分做一次"分段缓存"（将当前缓存保存一个快照）
- 超过 10 分钟的录音，在浮动条显示温和提示："录音已超过 10 分钟，建议分段处理以获得更好效果"
- 支持"分段转写"：对于超长录音（>5分钟），自动分段（按静音间隔分割）分别转写后拼接

### 5.3 转写历史管理

```
~/.tingyuxuan/
├── cache/
│   └── audio/           # 音频缓存文件
│       ├── 2026-02-24T143000_dictate_abc123.wav
│       └── 2026-02-24T145000_translate_def456.wav
├── history/
│   └── transcripts.db   # SQLite 数据库
│       表：transcripts
│       字段：id, timestamp, mode, raw_text, processed_text,
│             audio_path, status(success/failed/pending),
│             app_context, duration_ms, language
└── settings/
    └── config.json      # 用户设置
```

**历史记录界面**（从菜单栏进入）：
- 列表展示最近的转写记录
- 每条记录显示：时间、模式、处理后文本预览、状态
- 支持操作：复制文本、重新转写（使用缓存的音频）、删除
- 失败的记录高亮显示，支持一键重试
- 搜索功能：按关键词搜索历史转写内容

### 5.4 网络状态监控

- 应用启动时检测网络连通性（ping STT 和 LLM API endpoint）
- 网络断开时：
  - 菜单栏图标变为灰色/离线状态
  - 用户仍可按快捷键录音（音频本地缓存），但处理步骤会排队等待网络恢复
  - 网络恢复后自动处理排队的录音
- 定期（每 30 秒）静默检测网络状态

### 7.2 技术选型推荐

#### 核心引擎层：Rust

**推荐理由**：
- 跨平台编译：一套代码编译到 macOS、Windows、Linux、iOS、Android
- 高性能：音频处理、网络请求零开销抽象
- 内存安全：避免崩溃导致的音频数据丢失
- 与 Tauri 2.0 天然集成
- 通过 FFI 轻松与 Swift（macOS/iOS）和 Kotlin（Android）互调

**核心 Rust crate 推荐**：
- `cpal` — 跨平台音频采集
- `hound` — WAV 文件读写
- `reqwest` — HTTP 客户端（调用 STT/LLM API）
- `rusqlite` — 本地 SQLite 数据库
- `serde` / `serde_json` — 配置序列化
- `tokio` — 异步运行时
- `tauri` — Windows/Linux 桌面 UI 框架

#### UI 层：分平台策略

| 平台 | UI 框架 | 理由 |
|------|---------|------|
| **macOS** | **Swift + SwiftUI** | 原生体验最佳。Menu Bar 应用、浮动窗口、系统权限集成都需要原生能力。Rust 核心通过 Swift Package 或 C FFI 桥接 |
| **Windows** | **Tauri 2.0** | Rust 后端 + Web 前端（React/Solid），打包体积小（~10MB vs Electron ~150MB），性能好。System Tray 支持完善 |
| **Linux** | **Tauri 2.0** | 同 Windows，共享 Web 前端代码 |
| **iOS** | **Swift（Keyboard Extension）** | 必须原生开发自定义键盘。Rust 核心编译为 `.xcframework` 桥接 |
| **Android** | **Kotlin（IME Service）** | 必须原生开发输入法。Rust 核心编译为 `.so` 通过 JNI 桥接 |

#### 为什么不用 Electron？

| 维度 | Electron | Tauri 2.0 |
|------|----------|-----------|
| 打包体积 | ~150MB+ | ~10MB |
| 内存占用 | ~200MB+ | ~30MB |
| 启动速度 | 慢（加载 Chromium） | 快（原生 WebView） |
| 系统集成 | 中等 | 优秀（Rust 直接调系统 API） |
| 跨平台 | Win/Mac/Linux | Win/Mac/Linux/iOS/Android |

对于一个**常驻后台运行**的工具来说，内存和体积至关重要。Tauri 2.0 是更合适的选择。

#### 前端技术栈（Tauri Web 层）

- **框架**：React 18+ 或 SolidJS（更轻量）
- **样式**：Tailwind CSS
- **状态管理**：Zustand（轻量）
- **构建**：Vite

|------|------|---------|--------|
| **OpenAI** | 生态成熟，响应快 | gpt-4o-mini（日常）/ gpt-4o（高质量） | 首选 |
| **Anthropic Claude** | 长文本理解强，安全性好 | claude-sonnet（日常）/ claude-opus（高质量） | 首选 |
| **自定义 API（兼容 OpenAI 格式）** | 支持 Ollama、vLLM 等本地部署 | Qwen-2.5、DeepSeek 等 | 灵活性保障 |

> **关键设计**：听语轩必须支持用户自定义 API Base URL，以兼容所有 OpenAI API 兼容的服务（如 Ollama、LiteLLM、One API 等代理网关）。

## 八、开发模块拆解

### 模块 1：音频采集与缓存层

```
功能：系统级音频捕获 + 本地安全缓存
技术要点：
- 跨平台：Rust `cpal` 库统一音频采集接口
- macOS: AVAudioEngine 或 Core Audio（通过 Swift 桥接）
- 实时音频流 → PCM 数据 → WAV 文件写入
- 每 500ms flush 到磁盘（防崩溃丢数据）
- 静音检测 + 自动停止（可选）
- 噪音过滤/降噪前处理（可选，后续增强）
- 音频文件生命周期管理（创建/标记状态/清理过期文件）
- 提供实时音量数据给 UI 层驱动波形可视化
```

### 模块 2：语音识别层（STT）

```
功能：将音频转为原始文本
技术实现：
- 统一的 STTProvider 接口（trait），支持多种后端
- WhisperProvider: 调用 OpenAI Whisper API
- DeepgramProvider: 调用 Deepgram API（支持流式）
- AzureProvider: 调用 Azure Speech Services
- CustomProvider: 用户自定义 API endpoint
- 本地方案：whisper.cpp 集成（可选，离线运行）

接口设计：
  输入：音频文件路径 + 配置（语言提示、词典等）
  输出：原始转写文本 + 检测到的语言 + 置信度

错误处理：
  - 网络超时：返回 Error，触发重试机制
  - 音频过长：自动分段（按静音点分割）分别转写
  - API 错误：分类处理（限流/认证失败/服务不可用）
```

### 模块 3：LLM 后处理层

```
功能：智能润色、格式化、上下文适配
技术实现：
- 统一的 LLMProvider 接口（trait），支持多种后端
- OpenAIProvider: 调用 OpenAI Chat Completions API
- ClaudeProvider: 调用 Anthropic Messages API
- CustomProvider: 用户自定义 API（兼容 OpenAI 格式）
- 模块化 prompt 管理器：
  - 基础 prompt：去填充词、修正口误、整理格式
  - 上下文 prompt：根据当前应用类型调整语气
  - 翻译 prompt：指定目标语言，要求本地化表达
  - AI 助手 prompt：执行用户指令，可调用工具（搜索等）
  - 编辑 prompt：接收选中文本 + 编辑指令

Prompt 模板示例（听写模式）：
"""
你是一个智能语音输入助手。请将以下语音转写的原始文本整理为清晰、规范的书面文字。

规则：
1. 去除所有填充词（嗯、啊、那个、um、uh、like、you know 等）
2. 如果用户在说话中途改口或修正，只保留最终意图的表达
3. 去除不必要的重复词句
4. 自动补充合适的标点符号
5. 如果用户说了"第一"、"第二"或"首先"、"其次"等顺序词，生成有序列表
6. 保持用户原本的表达意图和核心内容，不要添加额外信息
7. 当前应用：{app_name}，请适配相应的语气（正式/非正式）

{user_dictionary_hint}

原始转写文本：
{raw_transcript}
"""

Prompt 模板示例（翻译模式）：
"""
你是一个专业翻译助手。请将以下语音转写文本翻译为{target_language}。

规则：
1. 先按照听写模式的规则清理原始文本（去填充词、修正口误等）
2. 翻译应自然流畅，读起来像目标语言的母语者所写
3. 不要直译，而是追求意译和本地化表达
4. 保持原文的语气、情感和正式程度
5. 如有专业术语，使用目标语言中的标准译法

原始转写文本：
{raw_transcript}
"""

Prompt 模板示例（AI 助手模式）：
"""
你是一个智能助手。用户通过语音向你提出了以下请求。
请理解用户意图并提供有帮助的回复。

规则：
1. 如果用户要求搜索或查询最新信息，请使用联网搜索获取结果
2. 回复应简洁实用，直接给出用户需要的内容
3. 如果适合，使用 Markdown 格式（标题、列表等）组织回复
4. 回复将被直接插入到用户当前的编辑区域

用户请求：
{raw_transcript}
"""

Prompt 模板示例（语音编辑模式）：
"""
用户选中了以下文本，并通过语音给出了编辑指令。
请按照用户的指令修改文本，只输出修改后的结果。

选中的文本：
---
{selected_text}
---

用户的编辑指令：
{raw_transcript}
"""

错误处理：
  - LLM 失败时回退到原始转写文本
  - 支持流式响应（SSE），边生成边输出
```

### 模块 4：输出注入层

```
功能：将处理后的文本插入到当前应用的光标位置
```

### 9.1 核心处理接口

```typescript
// STT Provider 统一接口
interface STTProvider {
  name: string;
  transcribe(audioPath: string, options: STTOptions): Promise<STTResult>;
  testConnection(): Promise<boolean>;
}

interface STTOptions {
  language?: string;          // 指定语言或 "auto"
  prompt?: string;            // 词典提示（提高特定词汇识别率）
  temperature?: number;       // 控制识别的确定性
}

interface STTResult {
  text: string;               // 原始转写文本
  language: string;           // 检测到的语言
  duration: number;           // 音频时长（秒）
  segments?: STTSegment[];    // 分段结果（可选）
}

// LLM Provider 统一接口
interface LLMProvider {
  name: string;
  process(input: LLMInput): Promise<LLMResult>;
  processStream(input: LLMInput): AsyncIterable<string>;  // 流式
  testConnection(): Promise<boolean>;
}

interface LLMInput {
  mode: 'dictate' | 'translate' | 'ai_assistant' | 'edit';
  rawTranscript: string;           // STT 原始转写文本
  targetLanguage?: string;         // 翻译模式的目标语言
  selectedText?: string;           // 编辑模式下的选中文本
  currentApp?: string;             // 当前应用名称
  currentWindowTitle?: string;     // 当前窗口标题
  userDictionary?: string[];       // 用户自定义词汇
  personalizationHints?: string;   // 个性化上下文
}

interface LLMResult {
  processedText: string;           // 处理后的文本
  tokensUsed: number;              // 消耗的 token 数
}

// 录音会话管理
interface RecordingSession {
  id: string;                      // 唯一会话 ID
  startTime: Date;
  endTime?: Date;
  mode: 'dictate' | 'translate' | 'ai_assistant' | 'edit';
  audioPath: string;               // 本地音频文件路径
  status: 'recording' | 'processing' | 'success' | 'failed' | 'cancelled';
  rawTranscript?: string;          // STT 结果
  processedText?: string;          // LLM 处理结果
  error?: string;                  // 错误信息
  appContext?: string;             // 当前应用上下文
}
```

## 十二、架构师审视：待明确与可优化项

从资深架构师的角度审视本文档，以下是一个 AI 开发助手在拿到这份文档后可能遇到的问题和需要进一步明确的地方：

### 12.1 需进一步明确的技术决策

1. **音频格式选择**：WAV（无损但体积大）还是 WebM/Opus（有损但体积小 10x）？建议默认使用 Opus 压缩以节省磁盘和上传带宽，同时保留 WAV 作为高保真选项。

2. **STT 调用模式**：是等录音结束后整段上传（Batch），还是边录边传（Streaming）？Batch 更简单但延迟高；Streaming 体验好但实现复杂。建议 MVP 阶段用 Batch，P1 阶段迁移到 Streaming。

3. **LLM 调用的流式输出**：最终文本是等 LLM 完整返回后一次性插入，还是通过 SSE 流式逐字插入？逐字插入体验更好（用户能看到文字逐渐出现），但实现复杂度更高。建议至少在 AI 助手模式下支持流式输出。

4. **浮动条的窗口层级**：浮动条需要 always-on-top，但不能遮挡用户正在输入的区域。需要考虑智能避让逻辑——当光标在屏幕底部时，浮动条是否应该移到顶部？

5. **多显示器支持**：浮动条应该显示在当前活跃的显示器上，而不是固定在主显示器。

### 12.2 可优化的方向

1. **离线模式**：集成 whisper.cpp 提供完全离线的 STT 能力，搭配本地 LLM（如通过 Ollama 运行 Qwen-2.5），实现完全断网可用。

2. **端到端延迟优化**：目前管线是串行的（录音完 → STT → LLM → 输出）。可以考虑：在录音过程中实时流式送入 STT（减少等待），STT 和 LLM 的网络请求使用 HTTP/2 多路复用，LLM 使用流式响应边生成边输出。

3. **Prompt 版本管理**：不同版本的 prompt 效果可能差异很大。建议建立 prompt 版本管理系统，支持 A/B 测试和回滚。

4. **成本估算与限制**：用户使用自己的 API Key，但可能不清楚成本。建议在设置面板中显示估算的 API 消耗（基于历史使用量），帮助用户理解成本。

5. **安全性**：API Key 存储需要加密（macOS 使用 Keychain，Windows 使用 Credential Manager），不能明文存储在 JSON 配置文件中。

6. **单元测试策略**：核心引擎层（Rust）应有完整的单元测试覆盖；Prompt 效果应有回归测试集（输入→期望输出对照表）。

7. **日志与诊断**：需要结构化日志系统，用户可一键导出诊断日志用于排查问题。

### 12.3 给 AI 开发助手的实施建议

当把这份文档交给 AI 进行开发时，建议按以下顺序推进：

1. **先搭骨架**：创建项目目录结构、配置构建系统（Cargo.toml + Tauri 配置 / Xcode 项目）
2. **先通管线**：实现最小可用路径——硬编码一段音频 → 调通 Whisper API → 调通 LLM API → 在终端打印结果
3. **再做 UI**：在管线跑通后，再添加浮动条、菜单栏等 UI 元素
4. **再做异常**：在正常路径稳定后，逐一添加异常处理和兜底机制
5. **最后做锦上添花**：个性化、词典、历史管理等增强功能

## 十三、注意事项与风险

1. **macOS 权限审核**：辅助功能权限和麦克风权限需要在应用首次运行时引导用户授权。必须提供清晰的引导界面，解释为什么需要这些权限。
2. **Fn 键冲突**：macOS 原生听写功能和"更改输入法"也绑定了 Fn 键，需在首次使用引导中提示用户在系统设置中修改。
3. **API 成本**：STT + LLM 双重 API 调用，需在设置面板中提供成本估算参考。
4. **延迟优化**：从说完话到文本出现的延迟是核心体验指标，目标控制在 2-3 秒内。
5. **隐私合规**：面向国内市场需注意数据出境和《个人信息保护法》合规；支持用户选择国内 API 服务（如智谱、百度文心等）。
6. **音频缓存磁盘占用**：长时间使用会积累大量音频缓存，需要自动清理策略和磁盘占用提醒。
7. **macOS App 签名与公证**：需要 Apple Developer 账号，否则用户安装时会被 Gatekeeper 阻止。
8. **输入法兼容性**：模拟键盘输入时需测试与各种输入法（搜狗、百度、微信输入法等）的兼容性，特别是中文输入场景。
